UniFi MCP — Repo Orientation & Infrastructure Map

A uv-managed Python monorepo of MCP servers, a REST/GraphQL API, and a Cloudflare relay that let AI agents manage Ubiquiti UniFi controllers.

TL;DR

What it is: three MCP servers (network · 180 tools, protect · 59, access · 34) that expose UniFi controller operations to AI assistants, plus a non-MCP REST+GraphQL server (apps/api) and a cloud relay pair (packages/unifi-mcp-relay sidecar ↔ apps/worker Cloudflare Worker) so cloud agents can reach your LAN without open ports.

The shape: every server follows the same strict layering — thin tools/ wrappers → managers/ domain logic → unifi-core ConnectionManager → controller HTTPS. Shared MCP plumbing (permissions, preview-then-confirm, lazy tool loading) lives in unifi-mcp-shared. All state lives on the controller; this repo is deliberately stateless (the API server's SQLite is the one exception).

Read first: AGENTS.md (canonical rules) and docs/ARCHITECTURE.md, then trace one tool family in apps/network.

03Internal Dependency Graph

Dependencies flow strictly downward: apps import packages, packages never import apps, and apps never import each other. Two components are intentionally islands — the relay and the worker talk to the rest of the system only over the network at runtime.

Python import dependency runtime protocol only (no code dependency)

The request path inside every MCP server

The same five layers in every app — enforced by AGENTS.md. This is the single most useful mental model for the repo:

MCP Client — Claude Desktop / Code, LM Studio, automation → MCP protocol (stdio · streamable HTTP · SSE)
↓
FastMCP Server apps/<app>/src/…/main.py — transport dispatch, tool registration via permissioned_tool
↓
Tool functions tools/*.py — thin wrappers only: validate args, format the response dict, delegate. No business logic.
↓
Managers packages/unifi-core/src/unifi_core/<product>/managers/ — all domain logic; never import from tools
↓
ConnectionManager unifi-core — single instance per process; aiohttp session, auto-reconnect, controller detection, retry
↓
UniFi Controller — REST APIs; all state lives here, not in this repo

Singletons (server, config, managers) come from @lru_cache factories in each app's runtime.py — tools import them from there, never instantiate directly.

06Conventions & Gotchas

Tool response contractEvery tool returns {"success": true, "data": …} or {"success": false, "error": "…"}. Exceptions never escape a tool function; error messages name the failed operation. Mutations return requires_confirmation: true with a preview until called with confirm=true.
Three API surfaces, disjoint IDsV2 controller API (cookie auth, Mongo ObjectIDs, snake_case), Integration API (X-API-Key, UUIDs, camelCase), and Alarm Manager v2 (SuperAdmin only). IDs are scoped to a "tool family" — passing one family's IDs to another fails at the controller. Silent cross-API ID translation is banned.
Access dual-path authUnique to the Access server: an API-key client on :12445 for reads and a proxy session (cookie + CSRF) on :443 for mutations. Each tool declares which path it needs.
Singletons via runtime.pyAll shared objects come from @lru_cache factories. Tests must monkey-patch the factory before importing tool modules. Exactly one ConnectionManager per process.
stdout is sacredIn stdio mode stdout carries JSON-RPC — all logging goes to stderr, with lazy %s formatting. A stray print will corrupt the protocol stream.
No hardcoded configHosts, ports, credentials, and flags live in config/config.yaml with ${oc.env:VAR,default} interpolation or UNIFI_-prefixed env vars. Hardcoding them in Python is a hard ban.
Validation models are centralizedPydantic models live in packages/unifi-core/…/models/<domain>.py — never inline JSON-schema dicts in tool functions.
SuperAdmin blast radiusAlarm Manager v2 needs a SuperAdmin credential; on a combined UDM console that credential is broad. Managers must map its 403s to an actionable "requires SuperAdmin" error.
Worker is a monorepo guestapps/worker is excluded from the uv workspace and keeps its own npm graphs (CLI and worker are separate packages). It lives here so relay-protocol changes land in one PR.
No SSO accountsMCP servers authenticate with a dedicated local admin account without MFA — Ubiquiti SSO cloud accounts are unsupported for setup.

07Glossary

TermMeaning here
MCPModel Context Protocol — the standard by which AI clients discover (tools/list) and invoke (tools/call) tools.
Meta-tools / lazy modeDefault discovery mode: only *_tool_index, *_execute, *_batch, *_load_tools register at startup (~200 tokens vs ~5000); domain tools load on demand. eager mode registers everything.
Tool familyA set of tools whose IDs are mutually portable (same UniFi API surface). Cross-family ID use is prohibited; bridging requires an explicit unifi_resolve_* tool.
Preview-then-confirmEvery mutating tool defaults to confirm=false, returning a preview payload; the mutation only executes on confirm=true. Bypass mode auto-injects it.
ManagerA domain-logic class in unifi-core (e.g. FirewallManager) — the only layer that talks to the ConnectionManager.
Relay / WorkerRelay = the LAN sidecar (Python, outbound WSS). Worker = the Cloudflare gateway (TypeScript, Durable Object). Together they are the "Cloud Relay".
RelayObjectThe Durable Object holding location registrations (SQLite) and live relay WebSockets; routes tool calls and fans out read-only calls across locations.
uiprotectThird-party Python client (née pyunifiprotect) the Protect server uses for REST + websocket event streaming, and for reaching Alarm Manager v2 via api_request().
tools_manifest.jsonPre-generated per-app tool metadata (names, descriptions, annotations) used for lazy loading and the tool index — regenerate via Makefile when tools change.
Plugins / skillsplugins/ ships Claude Code / Codex / OpenClaw bundles (MCP server config + setup skills like unifi-network-setup); plugins/cross-product/ has multi-server skills (incident-investigation, security-patrol).
